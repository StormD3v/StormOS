# Non-Goals

This document defines what StormOS is intentionally NOT. By clarifying what we are not building, we protect the vision and avoid scope creep.

---

## StormOS is NOT a Generic Portfolio

**What this means:**
- StormOS is not a standard portfolio template
- StormOS is not a copy of popular portfolio designs
- StormOS is not a "one-size-fits-all" solution
- StormOS is not meant to be used by others as a template

**Why this protects the vision:**
- Generic portfolios fail to differentiate
- Templates lack personality and authenticity
- Copying others dilutes Storm's unique identity
- StormOS should feel like Storm, not like everyone else

**What we do instead:**
- Create a unique, immersive experience
- Reflect Storm's personality and philosophy
- Design specifically for Storm's needs
- Build something that cannot be easily replicated

---

## StormOS is NOT an Admin Dashboard

**What this means:**
- StormOS is not a data-heavy dashboard
- StormOS is not a tool for managing data
- StormOS is not a complex application interface
- StormOS is not a productivity tool

**Why this protects the vision:**
- Dashboards are functional and utilitarian
- StormOS should be experiential and emotional
- Complex interfaces distract from the portfolio purpose
- Admin patterns don't fit the portfolio use case

**What we do instead:**
- Focus on storytelling and presentation
- Keep interfaces simple and focused
- Prioritize emotional engagement over functionality
- Use navigation patterns suited for exploration

---

## Decision Framework

### When Considering a New Feature

Ask these questions:

1. **Does this serve the portfolio vision?**
   - If no, reject it
   - If yes, continue

2. **Is this documented in the requirements?**
   - If no, reject it or document it first
   - If yes, continue

3. **Does this align with the non-goals?**
   - If it conflicts with a non-goal, reject it
   - If it aligns, continue

4. **Is this necessary for the MVP?**
   - If no, defer it
   - If yes, continue

5. **Does this add unnecessary complexity?**
   - If yes, simplify or reject it
   - If no, proceed

### When to Reconsider a Non-Goal

Non-goals may be reconsidered if:

1. The vision fundamentally changes
2. The target audience shifts
3. New requirements emerge that conflict with non-goals
4. A compelling reason to change is documented

Process:
1. Document the reason for reconsideration
2. Explain how the vision has changed
3. Get approval from maintainers
4. Update this document
5. Communicate the change

## Conclusion

These non-goals protect the StormOS vision by preventing scope creep and maintaining focus. When considering new features, always refer to this document to ensure alignment with the vision.

**Remember:** Knowing what you are NOT building is as important as knowing what you ARE building.
